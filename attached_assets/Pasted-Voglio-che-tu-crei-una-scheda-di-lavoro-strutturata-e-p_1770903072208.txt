Voglio che tu crei una scheda di lavoro strutturata e permanente per la migrazione e consolidamento della nuova logica di gestione immagini.

La scheda deve essere creata come file Markdown dentro la cartella:

report/

Nome suggerito:

WORKFLOW_MIGRAZIONE_GESTIONE_IMMAGINI.md

Obiettivo della scheda

Creare un documento operativo che contenga:

Stato attuale del progetto

Logica canonica approvata (TestImageContainer)

Strategia di consolidamento

Ordine degli interventi

Rischi e punti di attenzione

Test da eseguire per ogni step

Stato di avanzamento

Registro aggiornamenti progressivo

La scheda deve diventare il riferimento ufficiale per tutte le modifiche future legate a:

Gestione immagini desktop

Gestione immagini mobile indipendente

Overlay configurabile

Migrazione hero esistenti

Stabilizzazione preview mobile in admin

Struttura obbligatoria del documento
1. Stato attuale del progetto

Descrizione sintetica della situazione corrente

Componenti coinvolti

Problemi risolti

Problemi ancora aperti

2. Logica canonica approvata

Descrivere in modo tecnico e preciso:

Stato zero (fit-to-width)

Zoom monodirezionale

Clamp dinamico

Overlay regolabile

Separazione desktop/mobile

Persistenza dati

Media Library come unica fonte immagini

Questa sezione non deve cambiare nel tempo (è la base teorica).

3. Strategia di consolidamento

Definire:

Creazione componente unico riusabile

Eliminazione progressiva delle logiche legacy

Nessuna duplicazione permanente

Nessuna patch temporanea che resti nel codice

4. Ordine operativo degli step

Definire chiaramente:

Step 1 – Consolidamento componente unico
Step 2 – Aggiunta gestione mobile indipendente
Step 3 – Migrazione hero test (menu)
Step 4 – Validazione performance + UX
Step 5 – Migrazione progressiva altre hero
Step 6 – Stabilizzazione preview mobile admin

Ogni step deve avere:

Obiettivo

Componenti coinvolti

Rischi

Test obbligatori

Criteri di accettazione

5. Registro aggiornamenti

Una sezione in fondo al file che dovrà essere aggiornata sistematicamente dopo ogni step con:

Data

Step completato

Problemi incontrati

Diagnosi tecnica

Test effettuati

Esito

Decisioni prese

Regole fondamentali

Nessuna modifica codice in questo step.

Solo creazione della scheda strutturata.

Il file dovrà essere aggiornato ad ogni intervento futuro.

La governance del codice deve essere rispettata:

niente doppie logiche permanenti

niente fix sovrapposti

niente regressioni nascoste
